## 3. Breaking-change list

For consumers of the 0.0.x line:

1. **cem-theme CSS generators no longer use the browser XSLT-1.0 runtime.** The
   generated `dist/lib/css/*.css` is stable, but the generator *templates*
   changed to `type="cem-ml; version=0.0"`. Anyone who forked/embedded the
   generator HTML must adopt the substrate (Option B).
2. **The browser XSLT-1.0 transform *engine* is retired from the `custom-element`
   adapter** (see ¬ß4), but **legacy HTML+XSLT templates still work** ‚Äî they are
   transpiled to CEM-ML and rendered on the substrate (`CemElementRuntime`), not by
   a browser XSLT processor (the package verifier fails if `XSLTProcessor` reappears).
   The converter (`cem-elements/src/lib/legacy-xslt/convert.ts`) covers the Tier 1/2
   surface the legacy demos use ‚Äî `<xsl:value-of>`/`{‚Ä¶}`, `<xsl:for-each>` (incl. the
   `exsl:node-set($var)/*` inline-variable idiom, unrolled), `<xsl:if>`/`<xsl:choose>`,
   `<xsl:variable>`, `<slot>`, `<attribute>`/`<slice>`, AVT, and the XPath function
   subset (`contains`, `not`, `translate`, `substring*`, `position`, `count`, ‚Ä¶).
   **Tier 3 standalone XSLT stylesheets are not converted** (push-model
   `<xsl:template match>`/`apply-templates`/`call-template`/`sort`, EXSLT
   `func:function`, `<msxsl:script>`) ‚Äî they emit a conversion diagnostic; author that
   logic in CEM-ML/CEM-QL. The separate `custom-element-v0` DOM-projection bridge
   (explicit `lang`) stays deprecated-but-functional, removed next major.
3. **Deep `dist/` imports are discouraged.** Import package export subpaths
   instead ‚Äî `@epa-wg/cem-theme/tokens/cem.tokens.json`,
   `@epa-wg/cem-theme/tokens/cem.tokens.ts`, `@epa-wg/cem-elements`. Debug-only
   token artifacts (`cem.tokens.intermediate.json`, `cem.tokens.resolved.json`)
   must not be imported.
4. **Versioned contracts now apply a run-mode disposition on ingest** (BR-VC-9).
   A data/security payload (snapshot/`datadom`, edge-render-state) whose schema
   MINOR is ahead of the consuming build is **rejected** in an application run
   (hydration falls back to a fresh render). Forward-version payloads that
   previously hydrated silently now reject ‚Äî configure `runMode` per host
   (`application` default; `development` tolerates).

## 4. Bridge-window support matrix

**Policy (2026-06-09; revised 2026-06-10): the native XSLT *engine* is retired, but
legacy HTML+XSLT is supported by transpiling it to CEM-ML on the substrate.** The
browser XSLT-1.0 transform processor is gone in 0.1.0 (the `custom-element` adapter
delegates to `CemElementRuntime`; the verifier blocks `XSLTProcessor` from
returning). Legacy HTML+XSLT templates run via the DOM‚ÜíCEM-ML converter on the same
engine as migrated templates ‚Äî a legacy sample and its CEM-ML twin render
identically. Tier 3 standalone XSLT stylesheets (push-model / EXSLT / `msxsl:script`)
are not converted. The older `custom-element-v0` DOM-projection bridge stays
deprecated, removed next major (FF-5 gated).

| Surface | 0.1.0 | next major |
| --- | --- | --- |
| CEM-ML/CEM-QL substrate (`type="cem-ml; version=0.0"`) | ‚úÖ recommended | ‚úÖ |
| Native browser XSLT transform engine (`XSLTProcessor`) | ‚ùå **retired** | ‚ùå |
| Legacy HTML+XSLT via DOM‚ÜíCEM-ML conversion (Tier 1/2: `value-of`/`for-each`/`if`/`choose`/`variable`/AVT + XPath subset) | ‚úÖ supported | ‚úÖ |
| Tier 3 standalone XSLT stylesheets (`apply-templates`/`call-template`/`sort`, EXSLT, `msxsl:script`) | ‚ùå not converted (diagnostic) | ‚ùå |
| `custom-element-v0` DOM-projection bridge (explicit `lang`) | ‚öÝÔ∏è deprecated, functional | ‚ùå removed (FF-5 gated) |
| `custom-element-v0` / `cem-ml-v0` deprecated form ids | ‚öÝÔ∏è scanned by FF-5 | ‚ùå removed |

## 6. Rollback plan

- **Consumer rollback:** pin the previous published version
  (`@epa-wg/*@0.0.x`). Because the legacy XSLT-only path is *deprecated, not
  removed* in 0.1.0, most consumers can stay on 0.1.0 and keep using the legacy
  surface during the bridge window ‚Äî rollback is only needed for an unexpected
  substrate regression.
- **Forward-version safety:** the FF-6 SemVer axes + the BR-VC-9 ingest
  disposition (¬ß3.4) make an incompatible newer-payload deterministically
  detectable rather than silently mis-rendered.
- **Publisher rollback:** `nx release` tags are per-version; deprecate a bad
  release with `npm deprecate` and re-point consumers to the prior tag (the
  generated `CHANGELOG.md` + git tag give the exact prior state).

## 8. Dry-run rehearsal findings (2026-06-12)

`nx release version`/`changelog --dry-run` + per-`dist/` `npm pack --dry-run`,
everything short of tag/publish. Two real issues found; one fixed here.

1. **Conventional-commits will NOT auto-bump to 0.1.0.** `nx release version
   --dry-run` (no specifier) resolved current `0.0.14` from the git tag and computed
   **no bump** ("No files would be changed") ‚Äî of 332 commits since `0.0.14`, only 5
   use a conventional `feat:`/`fix:` prefix. The release MUST pass an explicit
   specifier: `nx release version 0.1.0` (verified: cleanly bumps all four cem-group
   manifests to 0.1.0).

2. **The auto-generated CHANGELOG is unreliable.** Because ~327/332 commits are
   non-conventional, `nx release changelog 0.1.0 --dry-run` omits essentially all the
   0.1.0 work (engine/substrate/legacy-XSLT/FF gates) and mis-anchors its "from" ref,
   surfacing stale pre-`0.0.14` commits. **Recommendation:** treat the ¬ß2 highlights as
   the canonical 0.1.0 changelog (hand-curate the entry); do not ship the raw
   auto-generated one. (Longer term: adopt conventional commits, or tag a clean base.)

3. **FIXED ‚Äî internal dependency range.** `cem-components` pinned
   `@epa-wg/cem-theme: "^0.0.14"`, which `^0.1.0` violates (nx
   `preserveMatchingDependencyRanges` blocked the version step). Changed to
   `"workspace:^"` (matching its `@epa-wg/cem-elements` dep; nx rewrites it to the
   published version at release). Re-run dry-run: no warning, all four bump to 0.1.0.

4. **Pack contents clean** (post engine-migration counts) ‚Äî see ¬ß5.
