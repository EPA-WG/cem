## 3. Breaking-change list

For consumers of the 0.0.x line:

1. **cem-theme CSS generators no longer use the browser XSLT-1.0 runtime.** The
   generated `dist/lib/css/*.css` is stable, but the generator *templates*
   changed to `type="cem-ml; version=0.0"`. Anyone who forked/embedded the
   generator HTML must adopt the substrate (Option B).
2. **Deprecated legacy XSLT-only `custom-element` surface** (see ¬ß4). Still
   functional in 0.1.0.
3. **Deep `dist/` imports are discouraged.** Import package export subpaths
   instead ‚ÄĒ `@epa-wg/cem-theme/tokens/cem.tokens.json`,
   `@epa-wg/cem-theme/tokens/cem.tokens.ts`, `@epa-wg/cem-elements`. Debug-only
   token artifacts (`cem.tokens.intermediate.json`, `cem.tokens.resolved.json`)
   must not be imported.
4. **Versioned contracts now apply a run-mode disposition on ingest** (BR-VC-9).
   A data/security payload (snapshot/`datadom`, edge-render-state) whose schema
   MINOR is ahead of the consuming build is **rejected** in an application run
   (hydration falls back to a fresh render). Forward-version payloads that
   previously hydrated silently now reject ‚ÄĒ configure `runMode` per host
   (`application` default; `development` tolerates).

## 5. npm package-contents check

Dry-run `npm pack` findings (2026-06-09). **Important:** `custom-element` is
built to a self-contained `dist/` (its own `dist/package.json`, imports vendored
to `./vendor/@epa-wg/cem-elements/...`), and **publish is from `dist/`** ‚ÄĒ the
source dir's root `custom-element.js` still has the workspace-relative
`../cem-elements` import and is not the published entry. So pack from `dist/`,
not the source dir.

| Package | Publish root | Verdict |
| --- | --- | --- |
| `@epa-wg/cem-elements` | `dist` (`!**/*.tsbuildinfo`) | ‚úÖ clean (dist + subpath exports) |
| `@epa-wg/cem-theme` | `dist` (`!**/*.tsbuildinfo`) | ‚úÖ clean (dist + token subpath exports) |
| `@epa-wg/custom-element` | `dist/` | ‚úÖ 80 files / 2.9 MB, self-contained; no `.vscode`/dev cruft. **One stray:** `vendor/@epa-wg/cem-elements/dist/tsconfig.lib.tsbuildinfo` (38 kB) copied in by the substrate vendoring (`compile-html.mjs stageSubstrateRuntime`) |

(An earlier source-dir pack showed 152 files incl. `.vscode/` ‚ÄĒ that was the
**dev** dir, not the published artifact; the `files:["*"]` on the source manifest
is a non-issue for npm because the build emits a clean `dist/` with its own

**Pre-publish action (minor, optional):** drop the stray `*.tsbuildinfo` from the
vendored copy ‚ÄĒ exclude it in `stageSubstrateRuntime` (it already runs `tsc`
incrementally) or add a `dist/.npmignore`. Not a blocker. Always
`npm pack --dry-run` from `dist/` before tagging.

## 6. Rollback plan

- **Consumer rollback:** pin the previous published version
  (`@epa-wg/*@0.0.x`). Because the legacy XSLT-only path is *deprecated, not
  removed* in 0.1.0, most consumers can stay on 0.1.0 and keep using the legacy
  surface during the bridge window ‚ÄĒ rollback is only needed for an unexpected
  substrate regression.
- **Forward-version safety:** the FF-6 SemVer axes + the BR-VC-9 ingest
  disposition (¬ß3.4) make an incompatible newer-payload deterministically
  detectable rather than silently mis-rendered.
- **Publisher rollback:** `nx release` tags are per-version; deprecate a bad
  release with `npm deprecate` and re-point consumers to the prior tag (the
  generated `CHANGELOG.md` + git tag give the exact prior state).
