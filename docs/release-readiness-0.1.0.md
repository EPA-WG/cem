## 3. Breaking-change list

For consumers of the 0.0.x line:

1. **cem-theme CSS generators no longer use the browser XSLT-1.0 runtime.** The
   generated `dist/lib/css/*.css` is stable, but the generator *templates*
   changed to `type="cem-ml; version=0.0"`. Anyone who forked/embedded the
   generator HTML must adopt the substrate (Option B).
2. **Deprecated legacy XSLT-only `custom-element` surface** (see §4). Still
   functional in 0.1.0.
3. **Deep `dist/` imports are discouraged.** Import package export subpaths
   instead — `@epa-wg/cem-theme/tokens/cem.tokens.json`,
   `@epa-wg/cem-theme/tokens/cem.tokens.ts`, `@epa-wg/cem-elements`. Debug-only
   token artifacts (`cem.tokens.intermediate.json`, `cem.tokens.resolved.json`)
   must not be imported.
4. **Versioned contracts now apply a run-mode disposition on ingest** (BR-VC-9).
   A data/security payload (snapshot/`datadom`, edge-render-state) whose schema
   MINOR is ahead of the consuming build is **rejected** in an application run
   (hydration falls back to a fresh render). Forward-version payloads that
   previously hydrated silently now reject — configure `runMode` per host
   (`application` default; `development` tolerates).

## 4. Bridge-window support matrix

**Policy (2026-06-09): deprecate now, remove next major.** The legacy XSLT-only
path ships deprecated-but-working in 0.1.0 and is removed in the following major,
giving consumers one full major to migrate. The FF-5 removal gate enforces zero
in-repo consumers of the deprecated form before removal.

| Surface | 0.1.0 | next major |
| --- | --- | --- |
| CEM-ML/CEM-QL substrate (`type="cem-ml; version=0.0"`) | ✅ recommended | ✅ |
| Legacy XSLT-only templates (`custom-element` XSLT+XPath) | ⚠️ deprecated, functional | ❌ removed (FF-5 gated) |
| `custom-element-v0` / `cem-ml-v0` deprecated form ids | ⚠️ scanned by FF-5 | ❌ removed |

## 6. Rollback plan

- **Consumer rollback:** pin the previous published version
  (`@epa-wg/*@0.0.x`). Because the legacy XSLT-only path is *deprecated, not
  removed* in 0.1.0, most consumers can stay on 0.1.0 and keep using the legacy
  surface during the bridge window — rollback is only needed for an unexpected
  substrate regression.
- **Forward-version safety:** the FF-6 SemVer axes + the BR-VC-9 ingest
  disposition (§3.4) make an incompatible newer-payload deterministically
  detectable rather than silently mis-rendered.
- **Publisher rollback:** `nx release` tags are per-version; deprecate a bad
  release with `npm deprecate` and re-point consumers to the prior tag (the
  generated `CHANGELOG.md` + git tag give the exact prior state).
