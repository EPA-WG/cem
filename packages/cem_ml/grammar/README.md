# CEM-ML Editor Grammar Assets

This directory ships the editor-side grammar artifacts that mirror the
Rust tokenizer in [`../src/tokenizer/cem.rs`](../src/tokenizer/cem.rs).

| File                              | Purpose                                                                                  |
| --------------------------------- | ---------------------------------------------------------------------------------------- |
| `lexical.ebnf`                    | Machine-readable lexical grammar (Tier A). Source of truth for editor grammars.          |
| `cem-ml.tmLanguage.json`          | TextMate / VS Code syntax-highlighting grammar.                                          |
| `tree-sitter-cem/grammar.js`      | Tree-sitter grammar source for editor parse trees and the Rust parity gate.              |
| `tree-sitter-cem/src/parser.c`    | Generated by `tree-sitter generate`. Committed so downstream Rust builds need no CLI.    |
| `tree-sitter-cem/bindings/rust/`  | Rust binding crate (`tree-sitter-cem`) that compiles `parser.c` and exposes `LANGUAGE`.  |

## Synchronization

The Rust tokenizer is the canonical implementation. Three synchronization
gates apply:

1. **Token-kind cross-reference.**
   `lexical.ebnf` ends with a comment block listing every
   `SchemaTokenKind` variant. The Rust unit test
   `tokenizer::cem::tests::grammar_token_kinds_match_lexical_grammar`
   fails if a token kind is added in Rust but not listed in the EBNF.
2. **Tree-sitter parity (round trip).**
   `tests/tree_sitter_parity.rs` parses every
   `examples/cem-ml/**/*.cem` fixture in both engines, reduces each
   parse to a canonical sequence of structural events (directives,
   element opens / closes, attribute names and values, expression
   nodes, rich-content markers, presence of non-whitespace text), and
   asserts the two sequences are byte-for-byte identical. Run it with
   `cargo test -p cem-ml --test tree_sitter_parity`.
3. **Generated parser is up to date.**
   `tree-sitter-cem/src/parser.c` is generated. After editing
   `grammar.js`, regenerate via `nx run cem_ml:tree-sitter:generate`
   (or `tree-sitter generate` from the grammar directory) and commit
   the result alongside the grammar change.

## Tree-Sitter Build Instructions

The grammar's generated parser is committed, so editor extensions and
the Rust parity test do not need the tree-sitter CLI. Only contributors
modifying `grammar.js` need it:

```bash
# regenerate parser.c after grammar.js edits
nx run cem_ml:tree-sitter:generate

# or directly:
cd packages/cem_ml/grammar/tree-sitter-cem
tree-sitter generate

# sanity-check the generated parser against the canonical fixtures
for f in ../../../../examples/cem-ml/**/*.cem; do
  tree-sitter parse "$f" >/dev/null
done
```

Install the CLI either with `cargo install tree-sitter-cli` or
`npm install -g tree-sitter-cli` (npm binaries require glibc ≥ 2.39;
the cargo install works on older systems).

## TextMate Coverage

The TextMate grammar covers:

- Nodes (`{name ...}`) with element-name highlighting.
- Attributes (`@name=value`), namespaced (`@cem:action=primary`) and
  unprefixed.
- Content boundary (`|` / `▷`).
- Expression nodes (`{$ ...}`) — body delegated to the `source.cem-ql`
  scope so consumers can layer the cem-ql grammar on top.
- Rich-content enclosures (triple backticks).
- Line and block comments.
- Directives (`@doc`, `@ns`, `@default`, `@schema`).
- Quoted strings and cem-ql AVT spans in attribute values.

## Adding New Token Kinds

When adding a `SchemaTokenKind` variant:

1. Update `lexical.ebnf`'s production set and its token-kind cross
   reference block.
2. Update `tree-sitter-cem/grammar.js` to recognize the new surface
   form, then regenerate `parser.c` (see above).
3. Update `cem-ml.tmLanguage.json` if the surface form is visually
   distinct.
4. Re-run
   `cargo test -p cem-ml grammar_token_kinds_match_lexical_grammar`
   and
   `cargo test -p cem-ml --test tree_sitter_parity`.
